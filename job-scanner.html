Job Scanner

I am a research engineer who works on agents. So when I looked properly at my own job search, what I saw was not a chore. It was an agent problem with a live dataset attached: heterogeneous sources, unstructured text, a judgment call at every step and real money on every decision. So I built the system and scaled it up. It polls 630 companies every 10 minutes across 8 applicant tracking platforms, judges which roles are worth my time, writes a tailored resume for each one, checks its own output and then stops and waits for me. It has never sent an application on its own and it never will. I designed it that way on purpose.

The architecture

5 model stages, a cheap filter first and a human in the loop

The design is a funnel that spends as little as possible for as long as possible. A free local filter runs before any model is called. Then a cheap model triages. A stronger one judges and writes. A cheap one checks the work. Most postings die early and that is the point. Watch where the dots drop out.

A live posting, in flight and still a candidate.
Filtered out. Wrong country, wrong role or judged not worth my time. It never costs a resume.

Every stage is a separately scheduled job with its own lock. A slow tailoring run, which can take 20 minutes, never blocks the scanner from finding new postings. A dashboard shows a per stage health grid and backlog so I can see exactly where things are stuck.

The engineering

The hard parts were cost, ordering and trust

Prompting was never the difficult bit. Once a system has to run every 10 minutes forever, spend real money on every call and produce something I am willing to put my name on, the problems stop being about models and start being about architecture.

Cheapest filter first

A free local filter kills ineligible postings before a single model call happens. Earlier versions burned a full classification, sometimes a full tailoring pass, on jobs that were never eligible in the first place. Ordering the funnel by cost was the single biggest saving in the system.

2 models, split by what the work needs

Rule shaped work goes to a cheap model: triage and QA checking. Work that needs judgment goes to a stronger one: classification and writing. Using the strong model everywhere is too expensive to run forever. Using the cheap one everywhere produces resumes I would not send.

Prompt caching shaped the prompts

Every stage sends a byte identical system prompt so it hits the ephemeral cache. That constraint decided how the prompts are written: everything static lives in the system prompt, everything variable lives in the user turn. The cost structure drove the design.

The model checks the model

Nothing reaches me unreviewed. A QA stage inspects every generated resume for house style violations, filler phrases, unsupported skill claims and invented entries, then requeues anything that fails. Anything the system generates gets checked before it reaches me.

Cross document consistency

Each tailored resume writes a terse fact sheet. Before a new resume goes out to a company I have already applied to, the QA stage compares it against those earlier fact sheets, so I cannot contradict myself across 2 applications to the same employer.

Human in the loop, by design

The pipeline stops dead at review. I read every tailored resume and I send every application myself. The automation exists to find and prepare, never to decide and send. Handing that judgment to a model would make the output worse and make me accountable for something I never read.

Determinism where it beats a model

The PDF compile uses no model at all. Neither does the staleness check. Every platform has a JSON API, so an HTTP status check is strictly more reliable than asking a model to read a page. When a deterministic check works, I use it instead of a model.

Independent stages, atomic state

8 scheduled jobs, each with its own lock, all reading and writing shared queues. Writes are temp file then atomic rename and queue updates re-read then merge, because a plain overwrite loses whatever another process appended while this one sat waiting on a model call.

2 kill switches at different scopes

One flag pauses everything that costs money. A narrower one pauses only tailoring, so I can edit my base resumes without anything generating against content that is about to change. Scanning keeps running through both, because scanning is free.

The finding I did not expect

A naive check called 61% of live jobs dead

The pool of tailored, not yet applied postings kept growing and some of those jobs had quietly closed. So I built a daily check to ask each board whether a posting was still live. The obvious version of that check was badly wrong and working out why was the most useful thing I did on this project.

The bug

The first version asked "is this job still in the company's current listing?" That looked right and was catastrophic. The listing call only ever fetched the first 20 results, which is correct for the scanner's own job, since new postings sort to the top, but is a silent truncation for anything else. One company had 1,657 open roles. On the first full run the check flagged 363 of 594 live jobs as stale, a 61% false positive rate.

The fix and the second bug under it

Paginating properly, using the endpoint's own total rather than trusting page 1, fixed most of it. But a spot check found a posting the listing said was missing that still returned full data on a direct fetch by its own ID. The search index is not consistent between reads. So "missing" now triggers a second, independent confirmation before anything is trusted.

Of the 522 confirmed live, 139 were rescued by the second pass from being wrongly flagged by the first. The 41 inconclusive ones matter more than they look. A bot protection block is not evidence that a job is closed, so the system records it as unknown and surfaces it for me to look at rather than quietly guessing in either direction. I would rather the system flag what it cannot determine than guess wrong.

Ingestion

10 adapters and the 4 that fight back

6 platforms return a complete listing with full job text in a single call. 4 need a 2 stage fetch: pull the listing, then pull each posting's description separately. Every adapter was verified live against a real company before being trusted, rather than written from documentation and assumed to work.

Platform names are withheld rather than merely blurred. They are not in this page's source either. I intend to open source this system once I am settled in a role. Publishing it while I am still using it would be cutting the branch I am sitting on.

One platform has a quirk worth describing even without naming it. Its page size and total fields are unreliable and asking for more than about 7 results silently returns an empty array despite real data existing. That adapter pages by "did this response come back empty" and never trusts the metadata. 2 further adapters are written and working but sit at 0 companies, because every live tenant I checked on those platforms was Europe only and this search is US only. Writing an adapter then admitting it has nothing to point at beats pretending the coverage is real.

Why it exists

My job search was really an agent problem

The underlying shape is familiar. Heterogeneous sources with inconsistent APIs. Unstructured text that needs judgment rather than rules. A decision worth real money at every step. Generated output that has to be checked before anyone can trust it. A human who stays accountable for whatever goes out. That is the same shape as any serious agentic deployment, which is why I found it worth building properly instead of grinding through by hand.

What came out is a compound AI system in the strict sense: multiple models with different capabilities and costs, routed by task, chained through orchestration, with retrieval, structured generation, an evaluation gate, observability, failure handling and a human decision point that cannot be bypassed. The interesting problems were about cost, ordering, trust and handling a model that is confidently wrong. Those turned out to be the real problems.

The repository is private while I am still using it. I am happy to walk through the architecture, the dashboard or any of the decisions above in a conversation.
