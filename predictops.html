PredictOps

Every ML Engineer job description lists Docker, FastAPI, CI/CD, MLflow. Almost none of them list a novel model architecture. So this project deliberately uses a boring, well-understood tabular classification problem and puts all the depth into the ops loop around it: the training comparison, the serving contract, the container, a CI gate that actually blocks a bad model before it ships, registry promotion and drift detection. The system around the model is the point.

The pipeline

Train, compare, serve, containerize

A single sklearn Pipeline (impute → scale numeric → one-hot encode categorical → classifier) so preprocessing can never leak test data into training. Logistic Regression and XGBoost both go through 5-fold stratified cross-validation and whichever wins on CV ROC-AUC gets logged to MLflow and saved. Then a FastAPI service loads it on startup and serves it behind validated request and response models.

The simpler model actually won. Logistic Regression beat XGBoost on 5-fold CV ROC-AUC, 0.846 to 0.842 and held the edge on held-out test data too (0.842 vs 0.837). That's the real result of the comparison, not a foregone conclusion picked to look tidy. The pipeline just reports whichever model wins and serves that one.

The ops loop is the project

The churn model is plumbing. The contribution is everything after it: a CI gate that fails the build below 0.80 ROC-AUC, registry promotion, and drift detection wired into one make pipeline.

The simpler model won, and shipped

Logistic Regression beat XGBoost on cross-validated ROC-AUC, 0.846 to 0.842, so the simpler model is the one that got promoted. Not a foregone conclusion picked to look tidy.

A tiny serving image, on purpose

A multi-stage build trains and exports to ONNX, then serves with onnxruntime alone, no sklearn at runtime. That takes the image from about 1.17 GB to 343 MB.

Where it actually stands

All 7 phases shipped.

This project was built one working phase at a time, each committed separately, each with its own "why" before moving on. Here's exactly what's real, instead of rounding up.

PhaseStatus
0 · Repo scaffolddone
1 · Training + MLflowdone
2 · FastAPI servingdone
3 · Multi-stage Dockerdone
4 · CI eval gatedone
5 · MLflow registry + promotiondone
6 · Evidently drift detectiondone
7 · Wire the full loopdone
Phase 4 was the centerpiece. A GitHub Actions workflow lints, tests and then runs a model-quality assertion: ROC-AUC ≥ 0.80 or the build fails, the same way a broken unit test fails a build. Verified by shipping a deliberately degraded model, watching CI go red, then fixing it. That's the difference between "I trained a model" and "a bad model literally cannot reach production here."
Phase 5-6: registry promotion and real PSI, not a rounded-off drift score. The winning run gets registered in the MLflow model registry and promoted Staging → Production. Drift detection runs Evidently with PSI (Population Stability Index) as the actual per-column stattest, comparing a reference split against a synthetically shifted "current" split, tripping an alert when any feature's PSI exceeds 0.15.
Phase 3 revisited: multi-stage Docker that actually earns the name. The first pass copied the entire build environment (MLflow, Evidently, pytest, ruff) into the runtime image. Splitting serving-only dependencies out cut the served image from 1.66GB to 733MB, a 56% reduction, from an actual measured build, not an estimate.

Why this project exists

The model is the boring part on purpose

My other 2 projects are LLM research. This one is the breadth piece that sits next to them: a plain tabular classification problem (customer churn, never the headline) so nobody mistakes the domain for the point. The point is the ops loop, the part almost every portfolio project skips because it stops at the trained model. A pipeline that prevents leakage, a served API with real input validation and a health check, a container that only ships what it needs and a CI gate that actually fails a merge. That's the same instinct behind the other 2 projects, applied to the plumbing instead of the research question: build the plumbing that makes the system trustworthy, beyond the part that demos well.
