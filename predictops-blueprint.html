System design & Code map Visuals

2 live blueprints of PredictOps, a production MLOps pipeline built around the parts most projects skip. The first traces the stages of progression: the ops loop from data to a CI-gated, registry-promoted, drift-monitored service. The second is the code map: every file in the repo, and how one artifact on disk becomes the next.

Hover anywhere on a diagram and a lens magnifies it, so you can read every label without shrinking the picture. The moving dots are data in flight, coloured by what kind of data it is.

01System design, the stages in run order

Read it top to bottom. The model is deliberately boring, a tabular churn classifier; the point is everything after it. On the left are the inputs and config. The centre spine is the ops loop in run order: load, train and compare, export to ONNX, the CI eval gate, registry promotion, serving, and drift. The right shows what each stage writes to disk. The amber path is the drift monitor watching production for feature shift.

The ops loop is the project

The churn model is plumbing. The contribution is everything after it: a CI gate that fails the build below 0.80 ROC-AUC, registry promotion, and drift detection wired into one make pipeline.

The simpler model won, and shipped

Logistic Regression beat XGBoost on cross-validated ROC-AUC, 0.846 to 0.842, so the simpler model is the one that got promoted. Not a foregone conclusion picked to look tidy.

A tiny serving image, on purpose

A multi-stage build trains and exports to ONNX, then serves with onnxruntime alone, no sklearn at runtime. That takes the image from about 1.17 GB to 343 MB.

02Code map, every file and the data lineage

Same project, a different lens. The centre spine is the data lineage: a CSV becomes a cleaned frame, a trained model, an ONNX artifact, a promoted registry version, then a served prediction.
