The finding I keep coming back to

More monitoring is not always better

Add a critic and fixer to a weak coding agent and it jumps 32 points. Add the exact same layer to a strong one and it quietly loses ground. Every model I tested landed on the same downward line, with the crossover at 60 to 65 percent baseline. You need to know which side of that line your model sits on before you ship.

Monitoring benefit versus baseline capability

Each point is a model. X axis is how good it is on its own, Y axis is what the monitoring pipeline added or removed.

Why it matters

You cannot pick an evaluator from a clean benchmark

The leaderboard hides the gap. Logistic Regression and XGBoost scored almost the same on clean data. Under attack one fell to an 82.1 percent false preference rate and the other held near chance. Nothing on the benchmark told you which to trust.
Competence is not robustness. The DeBERTa fine-tune was the only model to clear chance on clean data at 58.7% accuracy. It also degraded the most under attack, 19.3 points. Being good at the task is what gives an adversary something to take.
Knowing where it breaks is half the value. A reliability system that helps weak models and hurts strong ones is only safe if you know the crossover. I treat that failure condition as a core result.

How I work

Reliability is the whole point, not a nice to have

Assume someone will break it

Clean accuracy is table stakes. I build expecting an adversary, then measure exactly how far the system bends before it gives.

Prove it, do not claim it

Local inference, MLflow tracking, pinned dependencies, confidence intervals. Every result can be rerun.

Ship it live

The scoring pipeline runs as a public app you can poke right now. I would rather show a running demo than describe one.
